# 新机器迁移就绪性审计与实现方案

## 1. 结论

EnvWeave 当前已经具备配置文件、软件包、Git、差异编辑和配置批量事务等基础模块，但还不是可以在新硬件上“一键重建环境”的完整恢复工具。恢复向导已将缺失软件准备和有依赖顺序的配置事务组合成两个连续阶段，配置可整体回滚并能在重启后处理未完成状态；两个阶段尚未共享一个可暂停、可取消和可续跑的持久任务日志。

开发期不再兼容早期仓库格式。下一阶段直接设计 `format_version = 2`，以恢复计划为中心重构清单，避免在错误的 v1 模型上继续叠加兼容逻辑。

## 2. 当前关键缺口

| 优先级 | 缺口 | 当前表现 | 必须补充的能力 |
|---|---|---|---|
| P0 | 恢复编排 | 软件包和配置分散在不同页面，只能逐项操作 | 新机恢复向导、阶段化计划、批量执行、实时进度、暂停/重试/续跑 |
| P0 | 新机器预检 | 只识别 Linux/macOS，不了解发行版、架构、桌面环境和包管理器状态 | `MachineFacts`：OS、发行版、版本、CPU 架构、桌面、Shell、HOME、权限和工具可用性 |
| P0 | 硬件相关内容隔离 | 驱动、微码、内核模块、网络和挂载配置可能被当作普通内容迁移 | `portable / review / machine-bound` 策略；新硬件默认跳过驱动、fstab、crypttab、hostname 和网络身份 |
| P0 | 系统级恢复 | `/etc`、`/Library` 只能采集，不能恢复 | 独立 `PrivilegeBroker`；Linux polkit/pkexec 或受限 helper，macOS Authorization Services；仅开放白名单目标和操作 |
| P0 | 批量事务 | 已有写前备份、持久日志、整批回滚、启动发现和写后校验；尚不能主动暂停后续跑 | 增加取消点、阶段检查点、重试和恢复点命名 |
| P0 | 目录应用策略 | 当前直接替换整个目录，可能删除新系统生成的内容 | `replace / merge / keep-existing` 策略、排除规则、逐文件预览和删除清单 |
| P1 | 非文件型设置 | GNOME dconf、macOS defaults、服务启用状态等不能靠复制文件可靠迁移 | 受信任的类型化适配器：dconf、defaults、systemd units、launchd、字体和扩展清单；仓库不得携带任意命令 |
| P1 | 跨平台/跨架构映射 | 同一应用在 macOS、Arch、Intel、Apple Silicon 上路径和包名不同 | 条件规则、平台目标映射、包别名、架构约束、用户可审阅的主机覆盖层 |
| P1 | 包管理器引导 | 新 macOS 可能没有 Homebrew/mas；Arch 可能没有 AUR helper 或图形提权 | 前置诊断、Homebrew 安装引导、mas 可选安装、AUR helper 引导、软件源/签名/账号状态检查 |
| P1 | 软件包计划 | 清单只记录 provider/kind/name，不能表达可选项、来源信任和硬件属性 | 软件角色、可选性、来源、依赖阶段、替代包、平台约束、失败策略和审计版本 |
| P1 | 应用进程协调 | 正在运行的应用可能覆盖刚恢复的配置 | 恢复前检测相关进程，提示退出；恢复后可选重载，配置校验失败则回滚 |
| P1 | 文件元数据 | 目录权限、owner/group、ACL、xattr、macOS 扩展属性没有完整建模 | 明确的元数据策略；普通用户文件保留 mode，系统文件由权限代理设置 owner/group；ACL/xattr 按白名单保存 |
| P1 | 敏感信息 | 已有路径与内容规则扫描、二次确认、备份 Git 排除和受限暂存；尚未提供加密存储 | 增加仓库级策略；后续提供 age 加密的独立 secrets 模块 |
| P1 | Git 冲突 | pull/rebase 冲突只能报错 | 冲突状态页、继续/中止 rebase、文本三方合并；恢复前必须处于干净且同步的提交状态 |
| P2 | 可观测性 | 没有持久化的任务历史和诊断包 | 结构化事件日志、耗时、执行命令摘要、脱敏错误、导出诊断、恢复报告 |
| P2 | 发布可用性 | CI 只做检查，没有签名发布、自动更新和干净系统验收 | macOS 签名/公证、Linux 包矩阵、校验和、更新通道、Arch/macOS 虚拟机端到端恢复测试 |

## 3. Manifest v2

v2 不再把每个条目仅表示为“source + target”。核心模型应包含：

- `application_id`：归属的软件或系统组件。
- `scope`：`user` 或 `system`。
- `adapter`：`filesystem`、`gnome-dconf`、`macos-defaults`、`systemd` 等编译进程序的受信任适配器 ID。
- `apply_strategy`：文件使用 `replace / merge / keep-existing`；适配器使用自己的类型化策略。
- `portability`：`portable / review / machine-bound`。
- `conditions`：OS、发行版、架构、桌面环境和所需软件包。
- `dependencies`：恢复前必须完成的软件包或配置项。
- `sensitive` 与 `secret_policy`。
- `exclude`：缓存、锁文件、套接字、机器 ID 和运行时状态。
- `validate`：引用内置校验器 ID，不接受仓库提供任意 Shell 命令。

软件包清单 v2 增加平台、角色（应用/工具/驱动/硬件）、可选性、可信来源、替代包和恢复阶段。驱动、微码、内核、引导器与硬件绑定包默认进入人工确认区。

## 4. 新机器恢复流程

1. **打开或克隆仓库**：校验清单、Git 状态和仓库完整性。
2. **机器预检**：识别系统、架构、桌面、Shell、权限、网络和包管理器。
3. **生成恢复计划**：把项目分为可直接迁移、需要映射、需要确认、不适用和阻塞五类。
4. **引导基础工具**：处理 Homebrew、mas、AUR helper、Git 凭据和权限代理。
5. **安装软件包**：先官方源和基础运行时，再第三方源、GUI 应用与扩展。
6. **退出相关应用**：检测可能覆盖配置的进程。
7. **恢复用户配置**：按依赖顺序执行适配器和文件策略，每阶段建立恢复点。
8. **恢复系统配置**：只执行用户明确确认的白名单操作；机器绑定项默认跳过。
9. **重载与验证**：重载服务/桌面设置，运行内置校验器并重新扫描差异。
10. **结果与续跑**：持久化成功、跳过、失败和回滚结果；重启程序后可从未完成步骤继续。

## 5. 实施顺序

### R1：恢复内核

- Manifest/Package Manifest v2。
- `MachineFacts` 和 preflight API。
- `RestorePlan`、步骤状态、依赖排序和本地任务日志。
- 恢复向导 UI，先支持 dry-run，不执行写入。

退出条件：在空 HOME 测试环境中生成确定、可解释且可序列化的完整计划。

### R2：安全批量恢复

- 文件目录三种应用策略与删除预览。
- 计划级备份、执行、停止、重试、续跑和整体回滚。
- 应用进程检测和恢复后验证。

退出条件：任意步骤注入失败后都能回到计划开始前状态。

### R3：平台适配器与权限

- Linux systemd、GNOME dconf 和系统文件权限代理。
- macOS defaults、launchd、PATH 和管理员权限代理。
- Arch/Homebrew/mas/AUR 引导与诊断。

退出条件：Arch + KDE/GNOME 和 macOS 干净虚拟机完成端到端迁移。

### R4：跨机器策略与产品化

- 主机覆盖层、平台/架构映射、硬件相关内容审阅。
- Git 冲突工作流、敏感内容扫描、诊断报告。
- 签名发布、自动更新和安装介质验收。

退出条件：新硬件迁移不会默认写入旧机器的驱动、设备 ID、挂载和网络身份，并能输出完整恢复报告。

## 6. 近期不应继续扩张的方向

- 在恢复编排完成前，不继续单纯增加大量知识库路径；路径数量不能替代可靠恢复。
- 不允许知识库或仓库提供任意 Shell 命令；高权限迁移必须使用编译进程序、参数受限的适配器。
- 不把整个 `/etc`、`~/Library` 或桌面状态目录作为一个普通目录替换。
- 不把“单项操作可用”标记为“新机器迁移闭环已完成”。
